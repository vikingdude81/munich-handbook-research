==========================================================================================
  NECROMANCY → AI: MAPPING MEDIEVAL CONJURATION TO MODERN AGENT ORCHESTRATION
  From the Munich Handbook (CLM 849) to GPU Clusters
==========================================================================================

  The Munich Necromancer's Handbook describes structured protocols for
  summoning, binding, constraining, and commanding non-human intelligences.
  Modern AI agent systems do exactly the same thing—with language models
  instead of demons, system prompts instead of incantations, and API calls
  instead of circles.

  This document maps each structural element of 15th-century necromantic
  practice onto its direct counterpart in modern AI agent architecture,
  drawing from 2,903 extracted entities across the Munich Handbook sources.

  Munich: Never conjure 'a spirit.' Always name it: Astaroth, Berith, Floron.
  Each name carries specific capabilities and constraints.

  AI: Never call 'a model.' Always specify: qwen3-coder-next for reasoning,
  qwen3.5-9b for fast tasks, nomic-embed for vectors. The 'name' IS the
  capability contract.

  PATTERN 2: GRADUATED AUTHORITY (escalation chain)
  ─────────────────────────────────────────────────────────────────────────────
  Munich: 1st conjuration → 2nd conjuration → Bond of Solomon → divine threat
  AI: standard call → retry with params → escalate to larger model → human review

  PATTERN 3: CONTAINMENT BEFORE SUMMONING (safety first)
  ─────────────────────────────────────────────────────────────────────────────
  Munich: ALWAYS draw the circle BEFORE speaking the conjuration.
  Never summon without containment. The circle exists before the spirit.

  AI: ALWAYS set system prompt / guardrails BEFORE sending user input.
  Never give the model the task before establishing constraints.
  The sandbox must exist before the inference.

  PATTERN 4: PARALLEL SPECIALIST DEPLOYMENT (fan-out)
  ─────────────────────────────────────────────────────────────────────────────
  Munich: Castle ritual invokes 10+ spirits, each responsible for
  one aspect (walls, towers, gates, furnishing). They work in parallel.

  AI: Fan-out task to multiple workers, each handling one subtask.
  focus_mode.py sends the same prompt to all workers, aggregates results.
  The castle IS the merged output.

  PATTERN 5: SPECIFIC OUTPUT FORMAT (apparitio control)
  ─────────────────────────────────────────────────────────────────────────────
  Munich: Conjurations specify HOW the spirit must appear:
  'in form of a man,' 'in clear voice,' 'in the mirror,' 'in the boy's palm.'
  The form is DICTATED, not left to the spirit.

  AI: response_format={'type': 'json'}, structured output schemas,
  few-shot examples that SHOW the expected output format.
  You dictate the output form, not the model.

  PATTERN 6: DISMISSAL PROTOCOL (resource cleanup)
  ─────────────────────────────────────────────────────────────────────────────
  Munich: Every conjuration ends with 'Licentia' — formal dismissal.
  Spirit must depart peaceably. Forgetting dismissal = haunting.

  AI: Every inference call ends with connection close, VRAM free,
  context clear. Forgetting cleanup = memory leak, orphaned processes,
  runaway costs. The licentia IS the finally{} block.

  PATTERN 7: INSCRIPTION = PERSISTENT STATE (memory)
  ─────────────────────────────────────────────────────────────────────────────
  Munich: Names inscribed on rings, swords, parchment PERSIST across
  experiments. The Ring of Solomon works for every conjuration.

  AI: Embeddings, vector databases, fine-tuned weights persist across
  sessions. The 'inscriptions' are your trained parameters.
